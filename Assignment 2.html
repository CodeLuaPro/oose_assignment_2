2Problem Description

Your application will simulate the founding and operation of a collection of towns that exchange goods via a growing network of railways.

Obtain a copy of TownInput.java. If using a language other than Java or Kotlin, you will need to translate the code (or else find a way to invoke it from your chosen language).

2.1Basic Operation

The basic foundation of the simulation should be an event loop, like this (in Java):

while(System.in.available() == 0)  // <-- actual Java code
{
    // Pseudocode
    show status;
    check if anything needs to be done _at the present time_;
    do them all (if any);

    // Sleep for 1 second -- actual Java code
    try
    {
        Thread.sleep(1000);  // milliseconds
    }
    catch(InterruptedException e)
    {
        throw new AssertionError(e);
    }
}

In other words:

  • Loop until the user presses Enter;
  • Perform one simulation step on each loop iteration;
  • Wait 1 second between steps.

Note

The method System.in.available() returns the number of bytes in the console input buffer. This is 0 until theres any console input, and the user must press Enter to send any console input to the program. Thus, this event loop runs until the user presses Enter.

We cant just write scanner.nextLine() (for instance), because this waits for the input while doing nothing, and we want to actually run the simulation in the meantime.

Apart from this, we dont need any console user input. Your application will take input from another source, as explained below.

2.2Simulation Input

Given the TownsInput class (to be left unmodified), you must create a TownsInput object, and then call nextMessage(). This returns a string (or null) indicating that something has just happened. It will supply different messages at different points in time, and there will often be multiple messages available at the same time. Thus, when you call it, you should call nextMessage() repeatedly until it returns null, which means there are no further messages at the present time.

Each valid message string consists of 3 parts, separated by spaces:

  1. A message type either town-founding, town-population, railway-construction or railway-duplication;

  2. A town name, consisting of a combination of letters and underscores;

  3. Either a positive integer (if the message type is town-related), or a second town name (if the message type is railway-related).

You do need to validate messages. TownsInput will return invalid messages from time to time!

Note

Treat TownsInput itself as a library, not as a template. Your program must work with the original, unmodified version of this file (and potentially with another marker-supplied version).

So

  • DONT add your own methods or other design elements to this file. This would be BAD.
  • DONT count this file towards your use of patterns, generics, or other aspects of the marking criteria.

2.3Messages and Simulation Logic

The simulation must treat 1 actual second as 1 simulated day.

The simulation starts with zero towns and railways. From time to time, new towns are founded, and the population of any existing towns fluctuates. These are signified by the town-founding and town-population messages produced by TownsInput, where the integer gives the initial or new population of the specified town.

(Towns do not have any spacial location or geometric size/shape. They simply exist.)

Each town produces 1 unit of goods for per person per (simulated) day. So, for instance, a town with 100 people will produce 100 goods per day, and hence 1000 goods over 10 days, if its population remains constant. Goods are all produced right at the start of each day.

Goods accumulate (are stockpiled) in the town if they cannot be transported, but ideally as many goods as possible should be transported by railway to other towns. Once goods are transported, they are consumed, and no longer exist.

(Goods are not stockpiled after transportation, and theres no need to route any single unit of goods across multiple railways. Each unit of goods simply disappears once transported across a single railway link.)

Initially (when founded), a town has zero railway links to other towns, and hence cannot initially transport any goods. From time to time, new railways are constructed and upgraded, as follows:

  • A railway-construction message indicates a new single-track railway will commence construction between two specified towns.

    The construction will take 5 days (5 seconds), and the railway wont be usable during this time.

    Once completed, a single-track railway can carry 100 goods per day, but only in one direction on any given day. However, each day, the direction reverses. Thus, if TownA transports goods to TownB on day n, then TownB cannot (on the same day) send goods back to TownA, but on day n + 1, the direction reverses, and TownB can transport goods to TownA.

    The initial direction may be arbitrarily chosen. The original message may say railway-construction TownA TownB, but the ordering of the town names is irrelevant (and so this message is exactly equivalent to railway-construction TownB TownA).

    (Railway links do not intersect one another; they are all independent.)

  • A railway-duplication message indicates a second track will commence construction between two specified towns, along an existing single-track railway.

    Again, the construction will take 5 days, but the existing railway will continue to function in one direction at a time.

    Once completed, a dual-track railway can carry 100 goods per day in each direction independently.

For any given town, on any given day, your simulation should attempt to transport as many of the goods, both produced and stockpiled, as possible. Goods can be transported on the same day that they are produced.

(Theres no need to be strictly, mathematically optimal about how you do this.)

2.4Console Output

Each simulated day (actual second), the simulation must display:

  • The day number;
  • All messages received on that day; and
  • A table of summary information, with one row per town. For each town, the table should show the following information, as it was at the very end of the day:
    • The town name;
    • The town population;
    • The number of completed single-track railways connecting to the town;
    • The number of completed dual-track railways connecting to the town;
    • The number of goods currently remaining in the stockpile;
    • The number of goods transported today.

For instance:

---
Day 45:

town-founding TownC 777
town-population TownA 999
railway-construction TownC TownA

TownA p:999 rs:5 rd:8 gs:1234 gt:950
TownB p:888 rs:4 rd:7 gs:987  gt:850
TownC p:777 rs:0 rd:0 gs:777  gt:0
...
---
Day 46:
...

Assume there are also a number of other towns present in this example!

You can make this fancier if you like, as long as its reasonably legible.

2.5File Output

On each simulated day (if any of the relevant information changes), the simulation must output a text file, called simoutput.dot, representing the structure of the town/railway network.

This file will, of course, be frequently overwritten over the course of the simulation, but it must be kept up-to-date.

Heres an example of the file format:

graph Towns {
    TownA
    TownB
    TownC
    TownD
    TownE

    TownA -- TownB [color="black:black"]
    TownA -- TownC
    TownB -- TownD [style="dashed"]
}

This format is compatible with GraphViz, a suite of tools for visualising graph structures. Specifically, for our purposes:

  • The file must start with graph, a name, and an open brace.

  • The file must end with a close brace.

  • Spacing and indentation arent important (provided you dont merge words together).

  • The town names must be listed first, one per line, then the railways, one per line.

  • Use the following syntaxes for railways:

    Table 1.
    Syntax Railway type
    TownA -- TownB [style="dashed"] Single-track under construction
    TownA -- TownB Single-track completed
    TownA -- TownB [style="dashed",color="black:black"] Dual-track under construction
    TownA -- TownB [color="black:black"] Dual-track completed

If you like, you can augment this with other features of GraphVizs DOT file format, to achieve more informative or better-looking output. However, you must adhere to the official format (dont just invent your own syntax), and your output must be reasonably within the spirit of the exercise.

Its recommended that you run one of GraphVizs commands (e.g., neato or dot) on the files you produce, both for interest sake, and to test your code:

$ neato simoutput.dot -Tpdf -o simoutput.pdf

(Note: its not essential that you actually run GraphViz.)
